# Flutter × Solana Ecosystem Maturity — June 2026

## 0. web3flutter.dev (starting point, with a disclosure)

[web3flutter.dev](https://www.web3flutter.dev/) ("Web3 Flutter HQ") covers wallet flows, MWA, SPL/Token-2022, Metaplex, Anchor interaction, and recommends `solana`, `solana_mobile_client`, `borsh`, and `coral_xyz`. Disclosure that matters for recommendation independence: the site is maintained by GitHub user **immadominion**, who is also the author of `coral_xyz` — i.e., it is the project owner's own ecosystem hub, useful as a map but not third-party validation. Everything below was verified directly against pub.dev API and GitHub source.

## 1. package:solana (Espresso Cash → Brij) — the load-bearing SDK

- **Version/maintenance**: `solana` 0.32.0+1, published **2026-04-10**, publisher `cryptoplease.com` (Espresso Cash), 97 likes, **1,066 downloads/30d**, 130/160 pana points ([pub.dev API](https://pub.dev/api/packages/solana)). The monorepo **moved**: `espresso-cash/espresso-cash-public` now redirects to [`brij-digital/espresso-cash-public`](https://github.com/brij-digital/espresso-cash-public) (302 stars, last push 2026-04-10 — same day as last publish). Cadence is real but slowing (~quarterly), and the company has rebranded/pivoted to Brij; treat as single-vendor risk.
- **Platforms**: Android, iOS, Windows, Linux, macOS, **web** (`platform:web`, `is:wasm-ready`); 0.32.0+1's only change was "support SubscriptionClient on web" — websockets work in Flutter web.
- **Versioned transactions: YES, fully.** Verified in source: [`transaction_version.dart`](https://raw.githubusercontent.com/brij-digital/espresso-cash-public/master/packages/solana/lib/src/encoder/transaction_version.dart) (legacy/v0 prefix-bit detection), [`compiled_message.dart`](https://raw.githubusercontent.com/brij-digital/espresso-cash-public/master/packages/solana/lib/src/encoder/compiled_message.dart) (`CompiledMessage.legacy` / `CompiledMessage.v0` with `addressTableLookups`, full v0 byte-level encode+decode), `Message.compileV0({recentBlockhash, feePayer, addressLookupTableAccounts})` and `Message.decompile(..., addressLookupTableAccounts:)` in [`message.dart`](https://raw.githubusercontent.com/brij-digital/espresso-cash-public/master/packages/solana/lib/src/encoder/message/message.dart), plus `AddressLookupTableAccount`/state deserializer in `encoder/address_lookup_table/`.
- **Deserialize + re-sign an externally built base64 tx (the Flash Trade path): YES.** [`signed_tx.dart`](https://raw.githubusercontent.com/brij-digital/espresso-cash-public/master/packages/solana/lib/src/encoder/signed_tx.dart): `SignedTx.decode(base64)` parses signature array + compiled message (legacy or v0), exposes `compiledMessage.toByteArray()`; sign it with `Ed25519HDKeyPair.sign(bytes)` and rebuild `SignedTx(signatures: [...], compiledMessage: ...)`, then `encode()`. Helpers `signTransaction` / `signV0Transaction` exist in `helpers.dart`. **Caveat found in source**: `_signCompiledMessage` carries a `// FIXME(IA): signatures must match signers in the message accounts sorting` — when re-signing API-built txs you must place each signature at the index of its pubkey in `accountKeys` yourself (`SignedTx.fromBytes` maps by index). Write a deterministic test for this.
- **RPC coverage**: ~51 methods in [`rpc/client.dart`](https://github.com/brij-digital/espresso-cash-public/tree/master/packages/solana/lib/src/rpc) — full standard surface (getAccountInfo, getMultipleAccounts, getProgramAccounts, getLatestBlockhash, sendTransaction, simulateTransaction, getFeeForMessage, etc.). **Missing: `getRecentPrioritizationFees`** (zero hits) — hand-roll a raw JSON-RPC call for priority-fee estimation. WebSocket subscriptions: `accountSubscribe`, `logsSubscribe`, `programSubscribe`, `signatureSubscribe`, `slotSubscribe`, `rootSubscribe` as Dart `Stream`s with auto-unsubscribe.
- **Borsh**: sibling packages `borsh` 0.3.3 (2025-10-26, 165 dl/30d) + `borsh_annotation` with build_runner codegen — adequate for hand-written account structs.
- **Anchor**: minimal. `lib/src/anchor/` contains only `AnchorInstruction.withDiscriminator`/`forMethod` (sighash of `namespace:method`) and an account-discriminator check. **No IDL→Dart codegen.** The only IDL tooling is [`coral_xyz`](https://pub.dev/api/packages/coral_xyz) 1.0.0-beta.9 (2026-03-28, 3 likes, **51 dl/30d**) + `coral_xyz_codegen` (0 likes, 26 dl/30d) — authored by the project owner ([Immadominion/dart-coral-xyz](https://github.com/Immadominion/dart-coral-xyz)). For throtl-engine, generating a typed client means using/maturing your own beta package or writing instruction builders by hand against `borsh` — budget for the latter as baseline.

## 2. Mobile Wallet Adapter from Flutter

- **Official position**: Solana Mobile's docs ([docs.solanamobile.com/flutter/overview](https://docs.solanamobile.com/flutter/overview)) state the Flutter SDK is "community-developed and maintained... not an official library," pointing to Espresso Cash's packages.
- **`solana_mobile_client`** 0.1.2 (published **2025-05-25**; prior release Oct 2023), publisher cryptoplease.com, 6 likes, 942 dl/30d, **Android-only**. API: `getCapabilities`, `authorize({identityUri, iconUri, identityName, cluster})`, `reauthorize`, `deauthorize`, `signTransactions`, `signMessages`, `signAndSendTransactions` ([source](https://raw.githubusercontent.com/brij-digital/espresso-cash-public/master/packages/solana_mobile_client/lib/src/mobile_wallet_adapter_client.dart)). **It pins `com.solanamobile:mobile-wallet-adapter-clientlib:1.1.0`** in [android/build.gradle](https://raw.githubusercontent.com/brij-digital/espresso-cash-public/master/packages/solana_mobile_client/android/build.gradle) — that is **MWA 1.x**: legacy `cluster` (not `chain`) authorization, **no `signIn` (Sign-In-With-Solana)**, no multi-account, no `cloneAuthorization`. Mitigations: MWA payloads are opaque bytes, so **v0 transactions sign fine** through it; MWA 2.0 wallets (Seeker's ships 2.0-compliant) negotiate down to legacy sessions, so authorize/sign flows still work on Seeker. SIWS can be approximated with `signMessages` + your own SIWS-formatted message, losing wallet-native SIWS UI.
- **`solana_wallet_adapter`** (merigo-labs) 0.1.5 — **abandoned since 2023-12-04** (221 dl/30d), as are merigo's `solana_web3` (0.1.3, Dec 2023) and `solana_wallet_provider`. Do not build on these.
- **New entrant — `solana_kit`** ([openbudgetfun/solana_kit](https://github.com/openbudgetfun/solana_kit)): a Dart port of `@solana/kit`, repo created **2026-02-15**, last push 2026-06-07. `solana_kit` 0.5.0 (2026-06-01); **`solana_kit_mobile_wallet_adapter` 0.4.1 (2026-06-03) implements MWA 2.0**: `authorize({chain, signInPayload})` with `sign_in_payload` (SIWS, `siws_message.dart`), `cloneAuthorization`, protocol-version negotiation (v1 vs legacy in `types.dart`), even wallet-side scenarios. Adoption is near-zero (2 stars, 1 like, 212 dl/30d) — promising but unproven; if you need SIWS/MWA 2.0, plan to **vendor/fork** it rather than depend on it surviving.
- **Seed Vault**: dApps never talk to Seed Vault directly — the wallet app does, via MWA; Seeker compatibility therefore comes free through MWA. Espresso's `solana_seed_vault` (0.2.0+1, **Oct 2023**, 5 dl/30d) exists only for wallet builders and is dormant.

## 3. Publishing a Flutter APK to the Solana dApp Store

No framework restrictions exist — the store accepts any **release-signed APK** (and PWAs wrapped as Android apps); Flutter output is a standard APK ([docs](https://docs.solanamobile.com/dapp-publishing/intro), [Helius guide](https://www.helius.dev/blog/publishing-solana-mobile-apps)). Process: Publisher Portal account → mint App NFT → [`@solana-mobile/dapp-store-cli`](https://www.npmjs.com/package/@solana-mobile/dapp-store-cli) (Node 18–21) with a Solana CLI signer keypair + portal API key. Assets: 512×512 icon, 1200×600 banner, ≥4 screenshots/videos at 1080p, ≤30-char short description ([listing guidelines](https://docs.solanamobile.com/dapp-publishing/listing-page-guidelines)). [Publisher policy](https://docs.solanamobile.com/dapp-publishing/publisher-policy): crypto trading is explicitly the store's territory; regulated financial services must carry "all documentation required by Applicable Law" (no proprietary KYC mandate, defers to jurisdiction). Unlike Google Play: no billing tax, no real-money-crypto ban, and no Play Console foreground-service-type review.

## 4. Background trading guardian (feasibility)

Healthy, mainstream Flutter territory: **`flutter_foreground_task`** 9.2.2 (2026-04-01, 576 likes, **107k dl/30d**) runs a persistent-notification foreground service hosting a Dart isolate — your websocket guardian. Android 14/15 constraint: `dataSync` type is capped at ~6h/day; a trading guardian should declare `specialUse` (FOREGROUND_SERVICE_SPECIAL_USE) — Play requires a declaration form, but dApp Store distribution sidesteps that review. **`workmanager`** 0.9.0+3 (2025-08-31, 2,405 likes, 68.5k dl/30d) only does deferrable ≥15-min periodic work — unsuitable for real-time, fine for reconciliation sweeps. **`firebase_messaging`** 16.3.0 (2026-06-01, 2.45M dl/30d) high-priority data messages as wake-up backstop for liquidation/loss-floor alerts (Doze-exempt but throttled if abused). Recommended: foreground service while a session PDA is live + FCM backstop + server-side guardian as the real safety net (the phone must never be the only thing protecting the loss floor).

## 5. Honest gaps — what you hand-roll vs TypeScript

1. **MagicBlock ER**: no Dart SDK exists (pub.dev search "magicblock" → nothing). TS/Rust only. Hand-roll: ER router/RPC endpoint switching, delegation-status checks, MagicBlock-specific RPC calls — all doable over the existing `solana` RPC/jsonrpc layer, but every helper in MagicBlock's TS SDK is yours to port.
2. **Pyth Lazer**: no Dart package (pub.dev "pyth" → nothing relevant). Hand-roll the Lazer websocket subscriber + payload parsing + ed25519 signature verification (Dart `cryptography` package covers verify).
3. **Flash Trade**: TS SDK only — hence the deserialize/re-sign path; `SignedTx.decode` covers it (Section 1), but PnL/position account decoding needs hand-written Borsh structs from Flash's IDL.
4. **Anchor codegen**: nothing production-proven; `coral_xyz` is your own beta (51 dl/30d).
5. **Priority fees**: `getRecentPrioritizationFees` absent from `package:solana` — raw JSON-RPC.
6. **Flutter web wallets**: MWA is Android-only; web build needs a hand-rolled `dart:js_interop` bridge to wallet-standard (every Dart option is dead since Dec 2023). Plan web as second-class.
7. **Concentration risk**: the entire stack rests on one pivoting company (Espresso Cash→Brij, last push 2026-04-10) plus one 4-month-old solo port (`solana_kit`). Pin exact versions, vendor what's critical, keep a fork-ready mirror.

**Bottom line**: `package:solana` 0.32.x genuinely covers the hard cryptographic/wire-format requirements (v0+ALT, base64 re-sign, subscriptions, web). The MWA story is split: battle-tested-but-MWA-1.x (`solana_mobile_client`) vs MWA-2.0-but-newborn (`solana_kit_mobile_wallet_adapter`). Everything protocol-specific (MagicBlock, Lazer, Flash) is a TypeScript-to-Dart porting tax you pay yourself.
