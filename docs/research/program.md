# Pinocchio vs Anchor for throtl-engine under MagicBlock ER — Research Brief (verified 2026-06-13)

## 1. Pinocchio state of the world

`pinocchio` is at **v0.11.2** (released 2026-06-09; 548k total / 235k recent downloads on crates.io), maintained under **anza-xyz** (https://github.com/anza-xyz/pinocchio). It is `no_std`, zero-external-dependency, zero-copy (`AccountView`/`Address` types since the 0.10/0.11 API rename — older tutorials using `AccountInfo`/`Pubkey` are for ≤0.9). Entry points: `entrypoint!`, `lazy_program_entrypoint!`, `no_allocator!`, `nostd_panic_handler!`, and `process_entrypoint` for custom fast paths (README).

**Production usage:** the flagship is **p-token**, the SPL Token reimplementation now living in https://github.com/solana-program/token (`pinocchio/` dir; febo/p-token archived), slated to replace `Tokenkeg...` via feature gate under **SIMD-0266** after audits + Runtime Verification formal verification (https://www.helius.dev/blog/solana-p-token). MagicBlock itself ships `ephemeral-spl-token` (e-token) built on `ephemeral-rollups-pinocchio` (GitHub code search), and Turbin3's Accelerated program teaches Pinocchio+MagicBlock as a unit.

**Audit posture — be honest here:** Pinocchio was audited by **Zellic together with p-token** (report dated 2025-06-30, PDF in https://github.com/Zellic/publications: "Pinocchio and p-token - Zellic Audit Report.pdf"; second report "Anza p-token", Oct 2025). Findings: **8 findings, 1 critical** (OOB read via `?Sized` bound on `copy_val`), 3 high (missing `unsafe` on `Transmutable`/`Log`, UB in `AccountInfo::assign`, `MaybeUninit` invariant). Zellic's verdict verbatim: *"the project uses a not-insignificant amount of unsafe... while a particular commit could be determined to be safe, this is a weak assurance and is more likely to be broken with further changes... The use of the unsafe functionality by consumers of these libraries could result in the accidental introduction of subtle vulnerabilities."* Fixes landed post-audit; the library is far better-vetted than a year ago (fuzzing + formal verification of p-token), but **your program code inherits an unsafe-heavy, manually-checked style that auditors price in**.

**Benchmarks (real numbers):** p-token vs spl-token CUs (Helius): transfer **79 vs 4,645**, transfer_checked 111 vs 6,200, mint_to 123 vs 4,538 (~95% CU reduction); binary 95KB vs 131KB. Helius's Anchor→Pinocchio worked example: **649 → 109 CU** (https://www.helius.dev/blog/pinocchio). **I built both MagicBlock counter examples locally (solana-cli 3.1.10, anchor-cli 1.0.x)**: `anchor-counter` (anchor-lang 1.0.2 + ephemeral-rollups-sdk 0.14.3, features=["anchor"]) → **311,632-byte .so**; `pinocchio-counter` (ephemeral-rollups-pinocchio 0.14.4) → **57,128-byte .so**. Same delegate/commit/undelegate functionality, **5.45× size difference** (https://github.com/magicblock-labs/magicblock-engine-examples).

**Testing stack:** **mollusk-svm 0.13.1** (anza-xyz/mollusk; instruction-level SVM harness, `MolluskComputeUnitBencher` for CU regression tracking, fixtures, instruction chains) and **litesvm 0.12.0** — both framework-agnostic (they execute compiled `.so`). The official pinocchio-counter ships deterministic Rust tests using `solana-program-test` with a committed `dlp.so` (delegation program) fixture — that pattern works for testing delegate/undelegate against the real DLP locally.

**IDL/clients for non-Anchor programs:** `shank` 0.4.8 (metaplex) annotations → Anchor-compatible IDL; `codama` Rust crate **0.9.2** (2026-04-27) derives Codama IDL directly from Rust. Official Codama renderers: **JS and Rust only** (`@codama/renderers-js` 2.2.0, `@codama/renderers-rust` 3.1.0). **Dart is NOT "none", but it's thin:** (a) `codama-renderers-dart` 0.4.1 on npm (created 2026-02-28, single maintainer ifiokjr, targets the brand-new `solana_kit` 0.5.0 Dart SDK — immature); (b) **`coral_xyz` 1.0.0-beta.9 + `coral_xyz_codegen` on pub.dev — maintained by you (github.com/Immadominion/dart-coral-xyz)** — "universal Dart client for Solana programs — Anchor, Quasar, and Pinocchio... dynamic IDL-based program interactions". For 6 instructions with u8/u64 args, hand-rolling Dart instruction encoders is ~2 hours; IDL tooling is a non-factor in this decision.

## 2. Deploy cost math (formula verified against `solana rent` CLI)

`minimum_balance(n) = (128 + n) × 3480 lamports/byte-year × 2.0 years = (128+n) × 6960 lamports`. So 0.00000348 SOL/byte-year and ~0.00000696 SOL one-time per byte is **exact**, not approximate (verified: 165 bytes → 2,039,280 lamports). Upgradeable-loader deploy rents TWO accounts: program (36 B → 0.00114144 SOL) + programdata (45 B metadata + binary). **Current agave CLI defaults `max_len = program_len` exactly** (the old 2× allocation is gone; programdata auto-extends on upgrade) — verified in agave `cli/src/program.rs` (`let program_data_max_len = ... else { program_len }`, https://github.com/anza-xyz/agave/blob/master/cli/src/program.rs).

At **SOL = $67.15** (CoinGecko, 2026-06-13):
- Anchor ER counter (311,632 B): 2.17016 + 0.00114 = **2.1713 SOL ≈ $145.80**
- Pinocchio ER counter (57,128 B): 0.39881 + 0.00114 = **0.4000 SOL ≈ $26.86**
- **Difference ≈ 1.77 SOL ≈ $119, one-time and recoverable** via `solana program close`. Your throtl-engine will be larger than a counter in both cases, but the ratio holds (~5×). This is real money but not decision-grade for a production app.

## 3. THE CRITICAL QUESTION — ER without Anchor: YES, first-class and official

The ephemeral-rollups-sdk monorepo (https://github.com/magicblock-labs/ephemeral-rollups-sdk) contains a dedicated **`ephemeral-rollups-pinocchio` crate, v0.15.4 on crates.io (published 2026-06-12, version-locked to the main SDK)**. It is `#![no_std]` and provides everything the Anchor path has: `instruction::{delegate_account, delegate_account_with_any_validator, undelegate, commit, commit_and_undelegate, delegate_with_actions}`, **`intent_bundle::MagicIntentBundleBuilder`** (with a `no_vec` no-alloc variant; `.commit(&[acc])`/`.commit_and_undelegate(&[acc]).build_and_invoke(&mut buf)`), plus `acl` (permissioned ERs), `spl` (ephemeral ATAs), `vrf`, `crank`, `pda`. Caveat: it pins `pinocchio = "0.10"` while pinocchio is at 0.11.2 — match your program to the SDK's pinocchio version.

**What `#[ephemeral]` injects (read from `rust/ephemeral/src/lib.rs`):** exactly one Anchor instruction `process_undelegation(ctx: Context<InitializeAfterUndelegation>, account_seeds: Vec<Vec<u8>>)` + an `InitializeAfterUndelegation` accounts struct (`base_account, buffer, payer, system_program`, all unchecked), whose body CPIs `ephemeral_rollups_sdk::cpi::undelegate_account`. Its discriminator is the standard Anchor sighash — **I verified `sha256("global:process_undelegation")[..8] == [196,28,41,206,48,37,51,167]`** — and the official **pinocchio-counter example hand-implements precisely this**: it matches `UNDELEGATION_CALLBACK: [u8;8] = [196,28,41,206,48,37,51,167]` in its entrypoint and calls `ephemeral_rollups_pinocchio::instruction::undelegate(delegated_acc, program_id, buffer_acc, payer, ix_data)`, which checks `buffer.is_signer()` (DLP signs), parses the seeds vec from callback args, recreates the PDA and copies buffer data back (https://github.com/magicblock-labs/magicblock-engine-examples/blob/main/pinocchio-counter/src/entrypoint.rs). **Four official Pinocchio examples** exist (pinocchio-counter, pinocchio-ephemeral-permission-counter, pinocchio-private-counter, pinocchio-roll-dice) plus native `rust-counter`; MagicBlock's docs "Solana Program" how-to links the Pinocchio implementation as a peer of Anchor (https://docs.magicblock.gg/pages/ephemeral-rollups-ers/how-to-guide/rust-program). Mainnet ER validator identities (MAS1…/MEUGG…/MUS3…/MTEW…) and DLP `DELeGGvXpWV2fqJUhqcF5ZSYMS4JTLjteaAMARRSaeSh` are documented there.

## 4. Pyth Lazer inside the ER — no Anchor needed to consume

MagicBlock's oracle is its own Anchor program **`PriCems5tHihc6UDXDjzjeawomAwBduWMGAi8ZUjppd`** (https://github.com/magicblock-labs/real-time-pricing-oracle): a chain-pusher (identity `MPUxHCpNUy3K1CSVhebAmTbcTCKVxfk9YMDcUP2ZnEA`) writes Lazer updates every **50ms or 200ms per asset** into delegated PDAs (seeds `["price_feed","pyth-lazer",feed_id_string]`) shaped as `pyth_solana_receiver_sdk::price_update::PriceUpdateV2`. Your program just reads another account in the ER: check owner == `PriCems…`, optionally discriminator `[34,241,35,99,157,126,244,205]` (`sha256("account:PriceUpdateV2")[..8]`), then read **price: i64 at byte offset 73** (8 disc + 32 write_authority + 1 verification_level + 32 feed_id), conf u64 @81, exponent i32 @89, publish_time i64 @93 — the official docs themselves parse it by raw offset (https://docs.magicblock.gg/pages/tools/oracle/implementation). The `pyth-solana-receiver-sdk` crate is Anchor-based; in Pinocchio you skip it and read offsets (10 lines, zero CU waste). **No Anchor dependency for consumers.**

## 5. Verdict inputs (solo dev, 6 instructions, 1 delegated PDA)

- **Anchor (1.0.2, May 2026):** program in ~3-5 days; `#[ephemeral]`/`#[delegate]`/`#[commit]` macros do the ER plumbing; IDL falls out for free and your own `coral_xyz` Dart package consumes it; auditors quote Anchor programs at the low end because constraints are declarative. Cost: ~2.17 SOL deploy at counter-equivalent size (real throtl-engine likely 350-450KB ≈ $165-210), tick instruction ~5-15k CU (account deser + constraint checks on every 30-50ms tick).
- **Pinocchio:** the SAME architecture is officially supported with current, version-locked tooling and 4 reference examples — realistically **2-3× program-layer effort (2-3 weeks)** for a first Pinocchio program: manual discriminator dispatch, manual owner/signer/PDA checks (every missed check is a vuln class Anchor eliminates by default), `repr(C)` zero-copy state, hand-written Dart/TS encoders or shank/codama IDL. Tick instruction realistically <500 CU and ~60KB binary (~$27). Mollusk gives you per-instruction CU regression benches in CI either way.
- **Hysteresis reconciler / Flash Trade settlement is unaffected** by this choice — it happens via Magic Actions/L1 transactions outside the ER session, exactly as your architecture assumes.
- **Honest recommendation weighting:** the ER-without-Anchor blocker is gone (that was the only disqualifier). The remaining trade is $120-180 one-time rent + ~10k CU/tick (Anchor) vs ~2 extra weeks + permanently higher security self-discipline and audit scrutiny on unsafe-adjacent code (Pinocchio, cf. Zellic's warning). For a solo dev shipping a production perps app where the on-chain program guards real money, Anchor's constraint system is cheap insurance; Pinocchio is the optimization you do once the tick CU budget inside the ER measurably matters (MagicBlock's own e-token took that path). A pragmatic middle: Anchor with `zero_copy` state for the tick-hot PDA, with `mollusk` CU benches in CI to know exactly when/if a Pinocchio port pays.
