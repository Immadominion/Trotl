# MagicBlock ER from Dart/Flutter â€” research brief (verified 2026-06-13)

## 1. Magic Router: yes, a Dart client works â€” with a critical caveat

The Magic Router (devnet `https://devnet-router.magicblock.app`, mainnet `https://router.magicblock.app`, both also WSS on the same host â€” I verified a successful `101 Switching Protocols` upgrade) is a JSON-RPC 2.0 endpoint that inspects each transaction's **writable accounts** and routes it to the owning ER validator or to Solana L1 ([magic-router intro](https://docs.magicblock.gg/pages/ephemeral-rollups-ers/introduction/magic-router), [router API ref](https://docs.magicblock.gg/pages/ephemeral-rollups-ers/api-reference/er/introduction)). CORS is wide open: I curl-verified `access-control-allow-origin: *`, `access-control-allow-methods: *` on OPTIONS preflight and POST for both devnet and mainnet routers AND direct ER nodes â€” **browser/Flutter-web clients can hit it directly**.

**Caveat â€” it is NOT a full Solana RPC.** Empirical method matrix (curl, 2026-06-13): supported â€” `getAccountInfo`, `getBalance`, `getMultipleAccounts`, `getLatestBlockhash`, `getSignatureStatuses`, `getTransaction`, `getSignaturesForAddress`, `isBlockhashValid`, `getEpochInfo` (stubbed zeros), `sendTransaction`. **Method not found** â€” `getVersion`, `getHealth`, `getSlot`, `simulateTransaction`, `getMinimumBalanceForRentExemption`, `getFeeForMessage`, `requestAirdrop`, `getBlockHeight`, `getProgramAccounts`, `getTokenAccountsByOwner`. So pointing espresso-cash's `SolanaClient` (pub.dev [`solana` 0.32.0+1](https://pub.dev/packages/solana), Apr 2026) at the router alone will break (its `sendAndConfirm` helpers that call `getMinimumBalanceForRentExemption`/`simulateTransaction` fail). Custom router methods: `getRoutes`, `getIdentity`, `getDelegationStatus(pubkey) -> {isDelegated, fqdn, delegationRecord{authority,owner,delegationSlot,lamports}}`, `getBlockhashForAccounts([pubkeys]) -> {blockhash, lastValidBlockHeight}` â€” the last one returns the blockhash from the correct execution environment for the tx you're about to sign ([getBlockhashForAccounts](https://docs.magicblock.gg/pages/ephemeral-rollups-ers/api-reference/er/getBlockhashForAccounts), [getDelegationStatus](https://docs.magicblock.gg/pages/ephemeral-rollups-ers/api-reference/er/getDelegationStatus)). These are 10-line raw `http.post` calls in Dart; no SDK needed. The TS SDKs (`@magicblock-labs/ephemeral-rollups-sdk` and `-kit`, both 0.15.4, 2026-06-12) are thin wrappers over exactly this. Always send with `skipPreflight: true`, commitment `confirmed`; ER block time is 50 ms so "confirmed" lands in ~1 block. **ER rejects v0 transactions with address lookup tables** ([aperture/transaction_validation.rs](https://github.com/magicblock-labs/magicblock-validator/blob/master/magicblock-aperture/src/requests/http/transaction_validation.rs)) â€” use legacy txs for ER ops.

## 2. Dual-connection architecture (recommended primary)

`getRoutes` (verified live): devnet ER nodes `devnet-eu` (DEU) / `devnet-as` (SGP) / `devnet-us` (USA) `.magicblock.app`, mainnet `eu/as/us.magicblock.app`, all `baseFee: 0, blockTimeMs: 50`; plus TEE validators (`mainnet-tee.magicblock.app`, identity `MTEW...`). Validator identities are stable across networks: `MAS1...` (Asia), `MEUG...` (EU), `MUS3...` (US). You pick the region at **delegation time** by passing the validator pubkey (`DelegateConfig.validator`); thereafter the authoritative "where does this account live" is router `getDelegationStatus().fqdn` ([delegation lifecycle](https://docs.magicblock.gg/pages/ephemeral-rollups-ers/introduction/ephemeral-rollup), [dev-skill debugging runbook](https://github.com/magicblock-labs/magicblock-dev-skill/blob/main/skill/debugging.md)). MagicBlock's own AI dev-skill recommends: base RPC `https://rpc.magicblock.app/devnet|mainnet` for init/delegate, ER fqdn for delegated ops/commit/undelegate, router for status/discovery. Direct ER nodes implement a fuller RPC (`getVersion` returned `magicblock-core 0.12.3 / solana-core 3.1.12`, `getSlot` works) and WebSocket subs: `accountSubscribe`, `signatureSubscribe`, `programSubscribe`, `slotSubscribe`, `logsSubscribe` ([aperture/websocket](https://github.com/magicblock-labs/magicblock-validator/tree/master/magicblock-aperture/src/requests/websocket)) â€” espresso-cash's `SubscriptionClient` should work unmodified against `wss://<er-fqdn>`. For Throtl: subscribe to the session PDA on the ER fqdn for 30-50ms state echo; use the base connection for Flash Trade settlement txs. Ops status JSON: `https://status.magicblock.app/api/services`.

## 3. Session keys: skip the official program, roll your own

The MagicBlock session-keys program is `KeyspM2ssCJbqUhQ4k7sveSiY4WjnYsrXkC8oDbwde5`, a Gum protocol derivative now maintained at [magicblock-labs/session-keys](https://github.com/magicblock-labs/session-keys) (Anchor crate [`session-keys` 3.1.1](https://crates.io/crates/session-keys), May 2026). Layout (from `programs/gpl_session/src/lib.rs`): `SessionToken{authority, target_program, session_signer, valid_until: i64}` PDA at seeds `["session_token", target_program, session_signer, authority]`; V2 adds `fee_payer` and seed prefix `"session_token_v2"`. Instructions: `create_session(top_up: Option<bool>, valid_until: Option<i64>, lamports: Option<u64>)` (default validity 1h, **hard max 1 week**, optional 0.01 SOL default top-up of the session signer), `create_session_with_payer`, `create_session_v2`, `revoke_session` (V1: anyone can revoke, rent to authority; V2: anyone after expiry, authority-signed before, rent to fee_payer). Program-side it's the `#[session_auth_or(...)]` macro + `Session` trait ([integration docs](https://docs.magicblock.gg/pages/tools/session-keys/integrating-sessions-in-your-program)). The official client SDK is **React-only** (`@magicblock-labs/gum-react-sdk`, IndexedDB-encrypted keypair, web only â€” [installation](https://docs.magicblock.gg/pages/tools/session-keys/installation), [security](https://docs.magicblock.gg/pages/tools/session-keys/security)); there is no Dart SDK and **no MagicBlock package on pub.dev at all**.

**Verdict for Throtl: roll your own.** Since throtl-engine is a custom program with a per-session PDA anyway, storing `session_key: Pubkey` + `expires_at: i64` (+ loss_floor, max_exposure scope) in your own PDA and checking `signer == session_key && Clock::get()?.unix_timestamp < expires_at` is *exactly* what gpl_session's `validate()` does â€” minus one extra account, one CPI-adjacent dependency, and the question of whether the SessionToken (a base-layer account) clones cleanly into your ER session. It also lets you scope by *parameters* (loss floor) which gpl_session cannot. Prior art: Drift's `User.delegate: Pubkey` field ([protocol-v2 user.rs L75](https://github.com/drift-labs/protocol-v2/blob/master/programs/drift/src/state/user.rs)) is the canonical perps-DEX version of this pattern; MagicBlock's docs describe session keys generically as "expiry and access stored at the contract level" â€” your contract qualifies. No MagicBlock guidance forbids it. The session keypair is generated in Flutter (ed25519 via the `solana` package), held in memory/secure storage â€” Android Keystore on Seeker is strictly stronger than the browser IndexedDB model MagicBlock itself ships. Parent wallet signs ONE delegation+session-registration tx via MWA ([`solana_mobile_client` 0.1.2](https://pub.dev/packages/solana_mobile_client)); the session key signs every 30-50ms ER tx with zero base fee thereafter.

## 4. Cranks for the loss-floor guardian

Cranks = `MagicBlockInstruction::ScheduleTask{task_id: u64, execution_interval_millis: u64, iterations: u64, instructions: Vec<Instruction>}`, bincode-serialized, CPI'd from your program to `Magic11111111111111111111111111111111111111`; cancel via `CancelTask{task_id}` (authority-only). Crate: [`magicblock-magic-program-api` 0.12.2](https://crates.io/crates/magicblock-magic-program-api) ([crank docs](https://docs.magicblock.gg/pages/tools/crank/implementation), [crank-counter example](https://github.com/magicblock-labs/magicblock-engine-examples/tree/main/crank-counter)). The schedule tx must go to the **ER**, not base. Implementation reality ([task-scheduler.md](https://github.com/magicblock-labs/magicblock-validator/blob/master/docs/task-scheduler.md)): tasks live in an on-chain TaskContext account + a **node-local SQLite DB** (WAL mode); executions are built and signed by the **validator authority** as free ER txs; min interval configurable (`min-interval = "10ms"`); failures retry with backoff then park in a failed-task table; tasks are rehydrated/cleaned on validator restart. **No fee documented per execution** (ER txs are free on public nodes). **Honest gap: reliability is scoped to the single ER node** â€” there is no cross-node failover and no documented SLA, so a crank checking the loss floor every 100ms dies with the validator. Architecture consequence: crank = primary guardian; backstop = (a) `DelegateConfig.commit_frequency_ms` forcing periodic state commits to L1 (default is `u32::MAX` = never â€” you must set it), and (b) real margin/liq parameters on the Flash Trade L1 position so worst-case loss is bounded by exposure-since-last-commit.

## 5. Fees, lifetime, failure modes

Confirmed in docs **and** on-chain constants ([pricing](https://docs.magicblock.gg/pages/overview/additional-information/pricing), [dlp consts.rs](https://github.com/magicblock-labs/delegation-program/blob/master/dlp-api/src/consts.rs)): base fee 0/tx on public nodes (mainnet too, per live `getRoutes`); `SESSION_FEE_LAMPORTS = 300_000` (0.0003 SOL, charged at undelegation per delegation session); `COMMIT_FEE_LAMPORTS = 100_000` (0.0001 SOL per commit **after the first**); plus 10% of delegation-PDA rent on closure. Each delegation gets **10 sponsored commits**; lift via re-delegation or `magic_fee_vault` PDA + delegated fee payer ([dev-skill delegation.md](https://github.com/magicblock-labs/magicblock-dev-skill/blob/main/skill/delegation.md)). No hard account-lifetime limit is documented; lifetime is a delegation parameter. Validator failure: last committed state on L1 is always safe (fraud-proof committee per [whitepaper](https://arxiv.org/abs/2311.02650)); uncommitted ER state since the last commit is at the operator's mercy. `Undelegate` **requires the validator's signature** ([dlp undelegate.rs](https://github.com/magicblock-labs/delegation-program/blob/master/src/processor/fast/undelegate.rs)); the only escape hatch is admin-only `undelegate_confined_account` â€” i.e., the practical trust assumption is the MagicBlock operator, not permissionless exit. Mainnet ER is live in 3 regions + TEE; dedicated/self-hosted nodes: validator is open source ([magicblock-validator](https://github.com/magicblock-labs/magicblock-validator)), commercial support via development@magicblock.xyz; dedicated-node base fee modeled at 0.00000005 SOL/tx in the pricing simulator. Magic Actions (`MagicIntentBundleBuilder.add_post_commit_actions`, SDK 0.15.x) are the right hook for your hysteresis reconciler: commit ER session state + atomically fire the base-layer Flash settlement instruction ([magic-actions implementation](https://docs.magicblock.gg/pages/ephemeral-rollups-ers/magic-actions/implementation)) â€” note `CallHandler` actions pay via an escrow authority and run sequentially after the commit seals.

**Dart gaps summary:** no MagicBlock Dart SDK exists; you hand-roll (1) 4 custom router RPC calls, (2) delegate/undelegate instruction building (or expose them as your own program instructions, which the Anchor `#[delegate]`/`#[commit]` macros already do â€” your Dart client then only ever calls *your* program), (3) Anchor discriminators via sha256. All doc pages are fetchable as raw markdown (`https://docs.magicblock.gg/llms.txt` index) and the [magicblock-dev-skill](https://github.com/magicblock-labs/magicblock-dev-skill) is installable for CI agents.
